# テナント境界

テナントは「この AI エージェントや開発者に読ませてよい知識の集合」を分けるための境界です。
顧客案件、NDA、社外共有禁止、チームや開発者ごとの閲覧権限差のように、
混ぜてはいけない知識を分けるときに使います。

リポジトリごとの検索範囲は、テナントではなく `repo` メタデータで絞ります。
同じ AI エージェントが読んでよいリポジトリ群なら、リポジトリごとにテナントを
分ける必要はありません。

リポジトリ単位のテナントを標準にしない理由:

- 全体知識と個別リポジトリ知識を横断して検索しにくくなる
- モノレポ内の `apps/api`、`tools/review`、`docs/adr` のような単位を表しにくい
- リポジトリ名変更や分割統合が、アクセス制御の変更になってしまう
- テナント数が増え、MCP の読み取り設定やポリシーレビューが複雑になる

例外として、リポジトリそのものが顧客案件、NDA、社外共有禁止、開発者ごとの
閲覧権限差の境界である場合は、
そのリポジトリを専用テナントに登録しても構いません。
判断基準は「リポジトリ単位か」ではなく「読ませてよい相手が違うか」です。

推奨するテナント例:

- `secret-knowledge`
- `client-upwork-18384728-acme`
- `client-acme`

リポジトリやモノレポ内の領域で絞り込みたい場合はメタデータを使います。

```json
{
  "tenant": "secret-knowledge",
  "repo": "backend-testing-patterns",
  "path": "docs/e2e.md",
  "type": "doc",
  "tags": ["testing", "e2e"]
}
```

モノレポの場合も同じです。

```json
{
  "tenant": "secret-knowledge",
  "repo": "platform-monorepo",
  "path": "apps/api/internal/auth/session.go",
  "type": "code",
  "tags": ["api", "auth"]
}
```

MCP サーバーは、設定された読み取り境界の外にあるテナントが要求された場合、
その要求を拒否しなければなりません。
